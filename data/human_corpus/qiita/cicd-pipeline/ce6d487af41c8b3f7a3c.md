## 共通
### checkoutする
リポジトリをチェックアウトし、ワークフロー(Runner)がリポジトリにアクセスできるようにする。

```yml:cicd:yml
    steps:
      - uses: actions/checkout@v2
```

### self-hosted Runnerを指定する
自身で構築したRunnerでCICDを動作させる事ができる。

```yaml:ci.yml
jobs:
  build:
    runs-on:self-hosted
``` 

### 共通部分は再利用する（DRY原則を守る）
グローバルなenvを定義しそれを呼び出す事ができる。
（以下はEC2に`git pull`でDeployする事を想定したもの。）

```yaml:cicd.yml
jobs:
  deploy:
    env:
      SCRIPTS:
        echo '${{ secrets.SSH_KEY }}' > ssh_secret_key
        chmod 600 ssh_secret_key
        ssh -o StrictHostKeyChecking=no '${{ secrets.USER }}'@'${{ secrets.HOST }}' -i ssh_secret_key "
          cd /home/'${{ secrets.USER }}'/hogehoge/ && 
          git -c core.sshCommand=\"ssh -i ~/.ssh/deploy_key -F /dev/null\" pull
        "

    steps:            
      - name: Deploy
        run: ${{ env.SCRIPTS }}
```

## CIパイプライン構築で使うもの
### build成果物を保存する
GitHub Actionsの場合、buildの成果物を`artifacts`に保存する事ができる。

```yaml:ci.yml
    steps:
      - name: Archive UnitTest result of test
        uses: actions/upload-artifact@v2
        with:
          name: Unit Test Result
          path: |
            reports/test.html
            reports/coverage/
```

### パッケージ管理ツールを用いた環境構築
CI環境はまっさらな環境なのでUT・E2Eテストなどを流すには環境構築が必要。
その際、ローカルの時のように`npm install`や`pipenv install`は使わない。
CIでは`lockファイル`を用いて`npm ci`や`pipenv sync --dev`などで環境構築する。
