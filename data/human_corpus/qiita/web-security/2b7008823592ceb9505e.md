# 備忘録

- CORSについて
    - Cookieがない場合（＝認証がない）は、設定が甘くても大丈夫。
    - Cookieを使う場合（＝認証がある）は、厳しめの設定が必要。
    - `Access-Control-Allow-Credentials: true`, `Access-Control-Allow-Origin: <origin>`は必須の設定
    - デフォルトで`何でもあり`の設定になっている場合があるので注意
    - CORSの設定不備で、XSS脆弱性がなくてもなりすましができてしまう。※ただし、ローカルストレージを使用してリクエストヘッダーにトークンを付けている場合（ビアラー認証等）はなりすましはできない。

- JWTについて
    - `401 Missing or Invalid Authentication`
    - Firebaseのセッション管理を参考に
        - https://firebase.google.com/docs/auth/admin/manage-sessions
        - アクセストークンは取り消せません。
        - リフレッシュトークンは以下のいずれかの条件で無効化します。
            - ユーザ削除
            - ユーザ無効化
            - アカウント情報の変更があった（パスワード等）
            - ※リフレッシュトークンが無効化されれば、再ログインが必要になります
    - ログアウトどうするの？
        - 一般的には、トークンをクライアントから削除するだけ。⇨見かけのログアウト。
    - サーバー側でログアウトするには？
        - 有効期限を短くする　５分など　⇨ステートレス性が低下
        - ブラックリスト⇨ステートになってしまう
        - APIゲートウェイで管理する

- ステートフルトークンは？
    - トークンをデータベースで管理

- JWTを選ぶ理由
    - サーバでトークンのストレージ管理コストをなくすため
    - マイクロサービスが増えた場合、リソースサーバ単体での認可を実装しやすい

# JWTトークンの無効化テクニック
まずJWTペイロードに以下を用意する
- `sub` ユーザ識別子
- `iat` 発行日時
- `nbf` 開始日時（発行日時と同じ）
- `exp` 期限日時

ユーザのテーブルには以下の情報を用意する。
- `update_at` 更新日時

有効性の検証では毎回以下を確認する。
- `update_at` <= `iat` = `nbf` <= `exp` <= `現在日時`

※もしも、パスワード変更等で`更新日時:update_at`が更新された場合、その日時以前に作成されていたアクセストークンおよびリフレッシュトークンは無効となり、再ログインを要求されることになります。めでたしめでたし。
