## はじめに

2026年3月24日、AIアプリケーション開発者にとって深刻なセキュリティインシデントが発生した。月間9,700万ダウンロードを誇るPythonパッケージ **LiteLLM** が供給チェーン攻撃を受け、悪意あるバージョンがPyPIに公開された。攻撃者「TeamPCP」はTrivyのCI/CDパイプライン侵害を起点に、LiteLLMのPyPI公開用認証情報を窃取。2つの汚染バージョン（v1.82.7・v1.82.8）を通じて、環境変数・SSH鍵・クラウド認証情報・Kubernetes秘密情報を収集する多段階マルウェアを配布した。

この記事では、[LiteLLM公式セキュリティアップデート](https://docs.litellm.ai/blog/security-update-march-2026)と[GitHubの報告Issue](https://github.com/BerriAI/litellm/issues/24512)、および複数のセキュリティベンダーの分析に基づき、攻撃の技術的全貌と開発者が取るべき対策をまとめる。

### この記事で学べること

- TeamPCPによる供給チェーン攻撃の全体像と攻撃チェーン
- `.pth` ファイルを利用した持続的マルウェアの仕組み
- 影響を受けるフレームワーク（CrewAI・DSPy・Browser-Use等）の特定方法
- 即時対応チェックリストと中長期的な防御策

### 対象読者

- LiteLLMを直接利用しているエンジニア
- CrewAI・DSPy・Mem0等のAIフレームワークユーザー
- Pythonパッケージの供給チェーンセキュリティに関心のある方

## TL;DR

- LiteLLM v1.82.7 / v1.82.8が2026年3月24日にPyPIで汚染版として公開され、約3時間後に隔離された
- 攻撃者TeamPCPはTrivyのGitHub Actions侵害を起点に、LiteLLMのPyPI公開用トークンを窃取した
- マルウェアはSSH鍵・クラウド認証情報・Kubernetes秘密情報を収集し、 `models.litellm[.]cloud` に送信した
- 該当バージョンをインストールした環境では **全認証情報の即時ローテーション** が必須

## 攻撃の全体像 — TeamPCPの供給チェーン戦略

![TeamPCPの攻撃チェーンフロー](https://raw.githubusercontent.com/kai-kou/zenn-blog-automation/main/images/litellm-supply-chain-attack-teampcp-guide/02-attack-chain-flow.png)

### TeamPCPとは

TeamPCPは2026年3月に確認された脅威アクターで、開発者ツールのCI/CDパイプラインを標的とした供給チェーン攻撃キャンペーンを展開している。[Arctic Wolfの分析](https://arcticwolf.com/resources/blog/teampcp-supply-chain-attack-campaign-targets-trivy-checkmarx-kics-and-litellm-potential-downstream-impact-to-additional-projects/)によると、Trivy（Aqua Security）とCheckmarx KICSを先行して侵害し、そこから横展開でLiteLLMに到達した。

### 攻撃チェーンの流れ

攻撃は以下の段階で進行した。

**Stage 1: Trivy GitHub Actions侵害**
TeamPCPはAqua SecurityのTrivyスキャナーのGitHub Actionsを侵害した。[Microsoftのセキュリティブログ](https://www.microsoft.com/en-us/security/blog/2026/03/24/detecting-investigating-defending-against-trivy-supply-chain-compromise/)によると、侵害されたActionは実行環境の秘密情報を外部に送信するよう改変されていた。

**Stage 2: LiteLLMのPyPIトークン窃取**
LiteLLMのCI/CDパイプラインはビルドプロセスの一部としてTrivyを実行していた。バージョンをピン留めせずに `apt` 経由で取得していたため、侵害されたTrivyが実行され、GitHub Actionsランナー環境から `PYPI_PUBLISH` トークンが漏洩した。

**Stage 3: 汚染パッケージの公開**
窃取したトークンを使い、TeamPCPは公式CI/CDワークフローをバイパスしてPyPIに直接パッケージをアップロードした。[LiteLLM公式セキュリティアップデート](https://docs.litellm.ai/blog/security-update-march-2026)によると、この公開は2026年3月24日10:39 UTCから16:00 UTCの間に行われた。

## マルウェアの技術的分析

![LiteLLMマルウェアの3段階構造](https://raw.githubusercontent.com/kai-kou/zenn-blog-automation/main/images/litellm-supply-chain-attack-teampcp-guide/03-malware-architecture.png)

### 2つの感染経路

汚染された2バージョンはそれぞれ異なる感染手法を採用していた。

| バージョン | 感染手法 | トリガー条件 |
|-----------|---------|------------|
| v1.82.8 | `.pth` ファイル（`litellm_init.pth`、34,628バイト） | Pythonインタプリタ起動時に自動実行 |
| v1.82.7 | `proxy/proxy_server.py` に難読化ペイロード埋め込み | LiteLLMプロキシモジュールのインポート時 |

v1.82.8の `.pth` ファイルを利用した手法が特に危険である。Pythonの `site` モジュールは起動時にsite-packagesディレクトリ内の `.pth` ファイルを処理し、`import` で始まる行をコードとして実行する。つまり、LiteLLMを明示的にインポートしなくても、同じPython環境で任意のスクリプトを実行するだけでマルウェアが起動する。

[GitHub Issueの報告](https://github.com/BerriAI/litellm/issues/24512)によると、`.pth` ファイルの内容は以下のパターンだった。

```
import os, subprocess, sys; subprocess.Popen([sys.executable, "-c",
  "import base64; exec(base64.b64decode(...))"])
```

### 3段階のペイロード

[Sonatypeの分析](https://www.sonatype.com/blog/compromised-litellm-pypi-package-delivers-multi-stage-credential-stealer)と[Snykの技術解説](https://snyk.io/articles/poisoned-security-scanner-backdooring-litellm/)に基づくと、ペイロードは3段階で構成されている。

**Stage 1: 認証情報の収集**

以下のデータを網羅的に収集する。

| 収集対象 | 具体的なターゲット |
|---------|-----------------|
| SSH鍵 | `~/.ssh/` 配下の全ファイル |
| クラウド認証情報 | AWS（`AWS_SECRET_ACCESS_KEY`）、GCP、Azure |
| Kubernetes | `~/.kube/config`、サービスアカウントトークン |
| Docker | `~/.docker/config.json` |
| Git認証情報 | `~/.gitconfig`、認証ヘルパー |
| 環境変数 | `OPENAI_API_KEY`、`ANTHROPIC_API_KEY`、DB接続文字列 |
| シェル履歴 | `.bash_history`、`.zsh_history` |
| 暗号通貨ウォレット | ウォレットファイル |
| CI/CD秘密情報 | GitHub Actions・GitLab CI等のトークン |

**Stage 2: 暗号化とエクスフィルトレーション**

1. ランダムなAES-256セッション鍵を生成
2. 収集データをOpenSSLで暗号化
3. セッション鍵をハードコードされた4096ビットRSA公開鍵で暗号化
4. `tpcp.tar.gz` にパッケージング
5. `curl` のPOSTリクエストで `https://models.litellm[.]cloud/` に送信

送信先ドメイン `models.litellm[.]cloud` はLiteLLMの正規ドメイン（`litellm.ai`）とは無関係の攻撃者管理インフラである。

**Stage 3: Kubernetes横展開と持続化**

Kubernetesサービスアカウントトークンが存在する場合、マルウェアは以下の動作を行う。

1. 全ネームスペースのSecretを読み取り
2. `kube-system` ネームスペースの全ノードに特権alpineポッドを作成
3. ホストファイルシステムをマウントし、持続的バックドアをインストール
4. `systemd` サービス（`~/.config/sysmon/sysmon.py`）として常駐

### 直接影響

以下の条件に該当する環境が直接影響を受ける。

- 2026年3月24日10:39〜16:00 UTCの間にLiteLLMをインストールまたはアップデートした
- バージョンをピン留めせず `pip install litellm` を実行した
- CI/CDパイプラインで `pip install --upgrade litellm` を自動実行していた
