## はじめに

今回は **DTO（Data Transfer Object）** と **バリデーション** について解説します。

Todoの作成・更新APIを例にしながら、「DTOとは？」→「なぜDTOが必要なのか」→「DTO使い方」→「バリデーションを追加する」という流れで進めます。

## 1. Todoアプリのモデルを定義する
 
まず、Todoのデータ構造を定義します。
 
```ts
// models/tasks.model.ts
export enum TodoPriority {
  HIGH = 'HIGH',
  MEDIUM = 'MEDIUM',
  LOW = 'LOW',
}
 
export class TasksModel {
  id: string
  title: string
  priority: TodoPriority
}
```
 
Todoは `id`・`title`・`priority` の3つのプロパティを持ちます。
 
`enum`（列挙型）とは、**あらかじめ決まった値のリストを定義するもの**です。  
`priority` は `HIGH` / `MEDIUM` / `LOW` の3種類のみ許可する、という制約をコードで表現しています。  
これにより、想定外の文字列が `priority` に入ることを防げます。
 
 
## 2. コントローラーのエンドポイントを定義する
 
次に、このTodoを操作するためのAPIを作ります。  
まず `controllers/tasks.controller.ts` を作成し、以下のエンドポイントを用意します。
 
| 操作 | URL | HTTPメソッド |
|------|-----|-------------|
| 全件取得 | `http://localhost:3000/tasks` | GET |
| 1件追加 | `http://localhost:3000/tasks` | POST |
| 1件更新 | `http://localhost:3000/tasks/:id` | PATCH |
| 1件削除 | `http://localhost:3000/tasks/:id` | DELETE |
 
更新に `PATCH` を使うのは、`title` や `priority` の**一部だけを変更する部分更新**を行うからです。  `PUT` だと「リソース全体を丸ごと置き換える」意味合いが強いため、任意項目がある更新には `PATCH` を使うのが適切です。  
また、「どのTodoを更新するか」はURLの `:id` で指定します。`id` をボディに含めるのではなく、**URLで対象を特定するのがRESTの基本的な考え方**を使用しました。
 
コードにすると下記のようになります。
 
```ts
// controllers/tasks.controller.ts
import { Body, Controller, Delete, Get, Param, Patch, Post } from '@nestjs/common'
import { DeleteResult, InsertResult, UpdateResult } from 'typeorm'
 
import { TasksService } from '@/services/tasks.service'
import { TasksModel } from '@/models/tasks.model'
 
@Controller('tasks')
export class TasksController {
  constructor(private readonly tasksService: TasksService) {}
 
  // 全件取得
  @Get()
  findAllTasks(): Promise<TasksModel[]> {
    return this.tasksService.findAllTasks()
  }
 
  // 1件追加
  @Post()
  async addTask(@Body() input): Promise<InsertResult> {
    return this.tasksService.addTask(input)
  }
 
  // 1件更新
  @Patch(':id')
  async editTask(
    @Param('id') id: string,
    @Body() input,
  ): Promise<UpdateResult> {
    return this.tasksService.editTask(id, input)
  }
 
  // 1件削除
  @Delete(':id')
  async removeTask(@Param('id') id: string): Promise<DeleteResult> {
    return this.tasksService.removeTask(id)
  }
}
```
 
`@Post()` や `@Patch()` では `@Body()` を使ってリクエストのデータを受け取っています。  
しかし、この時点では `input` の型が `any` になっています。  
**「どんな構造のデータを受け取るのか」がまったくわからない状態**です。
 
→ ここでDTOの出番です。
 
 
## 3. DTOとは何か
 
**DTO（Data Transfer Object）** とは、「どんなデータを受け取るか」を定義するクラスです。
 
リクエストのデータの形・型を明示することで、次のメリットがあります。
 
- `input.title` などの**コード補完が効く**ようになる
- 後述する**バリデーションが使える**ようになる
 
### なぜ `interface` ではなく `class` を使うのか
 
TypeScriptには型定義の方法として `interface` もありますが、DTOには **`class`** を使います。  
理由は、後述する `class-transformer` が動作するのがクラスのみだからです。  
`interface` はコンパイル後に消えてしまうため、実行時に型情報を参照できません。
 
 
## 4. DTOを定義する
 
`dto/` ディレクトリを作成し、**作成用・更新用の2つのDTO** を定義します。  
操作によって受け取るデータの構造が変わるため、別々のクラスとして分けて管理します。
 
### 作成用DTO
 
```ts
// dto/create-task.input.ts
import { TodoPriority } from '@/models/tasks.model'
 
export class CreateTaskInput {
  title: string
  priority: TodoPriority
}
```
 
### 更新用DTO
 
```ts
// dto/update-task.input.ts
import { TodoPriority } from '@/models/tasks.model'
 
export class UpdateTaskInput {
  title?: string
  priority?: TodoPriority
}
```
 
更新の場合は `title` と `priority` を変更しないケースもあるため、`?` をつけて任意項目にします。
 
### コントローラーでDTOを使う
 
定義したDTOをコントローラーにインポートして、`@Body()` の型として渡します。
 
```ts
// controllers/tasks.controller.ts
import { CreateTaskInput } from '@/dto/create-task.input'
import { UpdateTaskInput } from '@/dto/update-task.input'
 
@Post()
async addTask(@Body() input: CreateTaskInput): Promise<InsertResult> {
  return this.tasksService.addTask(input)
}
 
@Patch(':id')
async editTask(
  @Param('id') id: string,
  @Body() input: UpdateTaskInput,
): Promise<UpdateResult> {
  return this.tasksService.editTask(id, input)
}
}
```
 
これで `input.title` や `input.priority` に型補完が効くようになりました。
 
**ただし、この時点ではまだバリデーションはされていません。**  
たとえば `title` に数値を渡しても、`priority` に想定外の文字列を渡してもリクエストは通ってしまいます。  
思わぬエラーを防ぐために、次はバリデーションを設定します。
 
 
## 5. バリデーションを設定する
 
### 必要なモジュールをインストール
 
```bash
npm i class-validator class-transformer
```
 
| パッケージ | 役割 |
|-----------|------|
| `class-validator` | DTOのプロパティにデコレータでバリデーションルールを定義する |
| `class-transformer` | リクエストのデータをDTOクラスのインスタンスに変換する |
 
### ValidationPipeを有効化する
 
`main.ts` に `ValidationPipe` を追加します。  
`useGlobalPipes` で設定することで、アプリ全体に一括適用されます。
 
```ts
// main.ts
import { NestFactory } from '@nestjs/core'
import { ValidationPipe } from '@nestjs/common'
import { AppModule } from '@/app.module'
 
async function bootstrap() {
  const app = await NestFactory.create(AppModule)
  app.useGlobalPipes(new ValidationPipe())
  await app.listen(3000)
}
bootstrap()
```
