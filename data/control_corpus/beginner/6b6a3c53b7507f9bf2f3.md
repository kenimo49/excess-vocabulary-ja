## はじめに

Durable Functions の上でエージェントを動かすとき、「エージェント同士のハンドオフ（HandOff）はどう実現すればいいのか」という疑問にぶつかることがあります。

OpenAI Agents SDK のような LLM フレームワークには HandOff がネイティブ機能として備わっており、`transfer_to_<agent_name>` というツール呼び出しひとつでエージェント間の制御移譲が完結します。
一方、Durable Functions にはそのような HandOff の概念はありません。

Durable Functions で AIエージェントを構築する場合、HandOff Workflow をネイティブ機能として備えている `Microsoft Agent Framwork` を利用することになりますが、Workflow を実行するには activity からじゃないと難しいため、せっかくの Durable Functions のセッション保持機能が使えません。

ではどう作るか。本記事では **External Event・Entity・Sub-orchestrator** という Durable Functions の基本部品を組み合わせることで、疑似的な HandOff と Human-in-the-Loop なチャットを同時に実現する設計を紹介します。

具体的には「OperatorAgent（トリアージ）」と「WeatherAgent（天気専門）」の2エージェントが協調して会話を継続するサンプルを使って、設計の意図とコードを順番に解説していきます。

## 1. Durable Functions / Durable Agent とは

### Durable Functions の基本コンポーネント

Durable Functions は Azure Functions の拡張機能で、**状態を保持したままの長時間ワークフロー**をサーバーレス環境で組める仕組みです。通常の Azure Functions は呼び出しごとに完全にステートレスですが、Durable Functions は実行の進捗をストレージにチェックポイントとして書き込み、関数が終了した後も状態を保って再開できます。

主な構成要素は次の4つです。

| コンポーネント | 役割 |
| -------------- | ------ |
| **Orchestrator** | ワークフロー全体の流れを定義する。何をどの順番で実行するかをコードで記述する |
| **Activity** | 実際の作業を行う単位。DB書き込み、API呼び出し、メッセージ送信など副作用を持つ処理を担当 |
| **Entity** | 軽量な状態管理ユニット。小さなオブジェクトの状態を永続化し、複数のオーケストレーターからアクセス |
| **Client** | Orchestrator の起動・クエリ・イベント送信などを担当する外部インターフェース |

Orchestrator の大きな特徴は **External Event（外部イベント）待ち** ができることです。

```python
# 外部からのイベントを最長1日待つ
user_input_task = context.wait_for_external_event("UserInput")
timeout_task = context.create_timer(deadline)
winner = yield context.task_any([user_input_task, timeout_task])
```

この `wait_for_external_event` を使うと、Orchestrator は処理を中断してユーザーの入力を待ち続けます。Consumption プランでは、この待機中には実行時間に対する課金は発生しないので、長時間の待ちを気にせず使えます（ただし再生処理やストレージトランザクションには別途コストがかかる点は注意が必要です）。

### Durable Agent とは

Durable Agent は、Durable Functions のワークフロー基盤の上に **エージェントのセッション継続** を乗せる仕組みです。`agent_framework_durabletask` ライブラリが提供する `DurableAIAgent` と `DurableAgentSession` を使うことで、エージェントとの複数ターンの会話をオーケストレーターの状態と一緒に永続化できます。

```python
# セッションが既存なら再接続、なければ新規作成
if topic.sessions.get("operator_agent"):
    session_id = topic.sessions["operator_agent"]
    thread = DurableAgentSession.from_session_id(
        AgentSessionId(name=session_id.name, key=session_id.key)
    )
else:
    thread = operator_agent.create_session()
    topic.sessions["operator_agent"] = TopicSessionId(...)
```

### 普通のチャットボットとの違い

Durable Agent と一般的なチャットボットは、同じ「エージェントとの会話」であっても設計の前提が大きく異なります。

| 観点 | 一般的なチャットボット | Durable Agent |
| ------ | ---------------------- | --------------- |
| 状態の保持 | アプリ側で会話履歴を管理 | Durable なワークフロー基盤に組み込まれる |
| 応答の返し方 | リクエスト → 即時レスポンス | 非同期実行。HTTP 202 + ポーリングが基本 |
| ストリーミング | 対応可能 | Orchestrator の出力はストリームできない※ |
| 長時間の停止・再開 | アプリ側で設計が必要 | Durable の仕組みで自然に扱える |
| 向いている用途 | インタラクティブなリアルタイム会話 | 状態保持が重要な非同期タスク・業務フロー |

※ Azure Functions（通常の HTTP Trigger）自体は v2 モデル + ランタイム 4.34.1 以降で HTTP ストリーミングをサポートしていますが、Durable の Orchestrator は起動直後に HTTP 202 を返して非同期ポーリングに移行する構造のため、ターン中の出力をリアルタイムにクライアントへ流すことができません。

## 2. Durable Agent で構築できるワークフローの種類

Durable Functions + Durable Agent で構築できるエージェンティックワークフローの種類を整理します。

| ワークフロー | 対応 | 実装方法 | 補足 |
| ------------ | ------ | --------- | ------ |
| **逐次実行** | ✅ ネイティブ対応 | Orchestrator で順番に `yield` | 公式ドキュメントで明示的にサポート |
| **並列実行** | ✅ ネイティブ対応 | `task_all` で複数タスクを並列スケジュール | Fan-out/Fan-in パターン |
| **Human-in-the-Loop** | ✅ ネイティブ対応 | `wait_for_external_event` + `raise_event` | 外部イベント待機で自然に実装できる |
| **HandOff** | ⚙️ 工夫が必要 | LLM の出力を解析してアプリ側で分岐 | 本記事で扱う方法。全履歴継承型の HandOff とは異なる設計 |
| **Group Chat** | ⚙️ 工夫が必要 | 別途フレームワーク（FastAPI上など）を組み合わせる | 本サンプルの構成ではネイティブに対応していない |

特に重要なのは **HandOff は「対応なし」ではなく「工夫が必要」** という点です。

OpenAI Agents SDK の HandOff の場合、エージェントが `transfer_to_<agent_name>` ツールを呼び出すと、ランナーが単一の `Runner.run()` 内で制御を移し、引き継ぎ先エージェントには**会話の全履歴が渡ります**。

本サンプルの疑似 HandOff は仕組みが異なります。LLM が返す JSON の `handoff_agent` フィールドをオーケストレーターが解析して分岐する方式で、各エージェントは独立した `DurableAgentSession` を持ちます。引き継ぎ時に渡るのは `response.message`（LLM の最後の応答テキスト）のみで、相手エージェントの会話履歴は引き継がれません。

**目的（エージェントへの処理委譲）は達成できるが、仕組みは別物** というのが正確な整理です。

## 3. Human-in-the-Loop をチャットに組み込む設計

### 設計課題と解法

ユーザーとの会話が複数ターン続く場合、毎回のメッセージを**現在会話中の正しいオーケストレーター**に届けなければなりません。

シンプルな実装では、毎ターンごとにトリアージエージェントを起動して「今どのエージェントと話しているか」を判断させる方法が思い浮かびます。しかしこれは推論コストとレイテンシが毎ターン上乗せされてしまいます。

本サンプルでは代わりに **Durable Entity（Topic）を使ったルーティング** を採用しています。Entity に「現在会話中のオーケストレーターの instance_id（`active`）」を保持し、HTTP Trigger がこれを読んで `raise_event` の送り先を切り替えることで、不要なトリアージを省いています。

    Operator["🤖 Operator Orchestrator\n(OperatorAgent)"]
