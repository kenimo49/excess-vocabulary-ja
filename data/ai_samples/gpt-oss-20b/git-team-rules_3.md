# Gitチーム運用ルール – エンジニアのための実践ガイド

## 1. はじめに  
Git は分散型バージョン管理であり、個人の作業だけでなく **チーム開発の基盤** です。  
しかし、ルールが曖昧だとブランチが増え過ぎて衝突が頻発し、レビューもスムーズに進まないことがあります。  
この記事では、実務で使える「Git チーム運用ルール」を整理し、コード品質・開発速度を両立するためのベストプラクティスを紹介します。

> **ポイント**  
> 1. ルールは「書く」より「守る」こと  
> 2. チーム全員で「決める」ことで所有感を高める  
> 3. 変更は「逐次的に」検証・デプロイ

---

## 2. ブランチ戦略（Git Flow をベースに）

| ブランチ | 目的 | ルール |
|----------|------|--------|
| `main` | 本番リリース | **読み取り専用**、マージは *protected* |
| `develop` | 次期リリースのベース | **自動ビルド** & **CI** が必須 |
| `feature/*` | 機能追加 | *PR* で `develop` へマージ、 `develop` からブランチを作る |
| `bugfix/*` | バグ修正 | *PR* で `develop` へマージ、 `develop` からブランチを作る |
| `hotfix/*` | 本番直近修正 | *PR* で `main` & `develop` へマージ、 `main` からブランチを作る |
| `release/*` | リリース準備 | *PR* で `main` & `develop` へマージ、 `main` からブランチを作る |

> **Tip**  
> ブランチ名は **Kebab-case**（例: `feature/user-auth`）で統一し、 `/*` は機能/バグ名を示す。

## 3. コミットメッセージ（Conventional Commits）

```
<type>[optional scope]: <subject>
[optional body]
[optional footer]
```

| タイプ | 意味 |
|--------|------|
| `feat` | 新機能 |
| `fix` | バグ修正 |
| `docs` | ドキュメント変更 |
| `style` | フォーマット変更 |
| `refactor` | リファクタ |
| `perf` | パフォーマンス改善 |
| `test` | テスト追加/変更 |
| `chore` | ビルド/ツール変更 |

**例**

```git
git commit -m "feat(user-auth): add OAuth2 support"
```

### ルール

1. **コミット単位で機能を完結** – 1 つのコミットは 1 つの機能/修正に限定。  
2. **短くて分かりやすい subject** – 50 文字以内で要点を表現。  
3. **body は必要に応じて詳細** – 変更理由・背景を書き、レビュアーが迷わないように。  
4. **footer に JIRA/Trello タスク番号** – 追跡しやすくする。

---

## 4. Pull Request (PR) のルール

| 項目 | ルール |
|------|--------|
| タイトル | `feat: <短い説明>` など、 `type` を入れた形式 |
| Description | *Why*、*What*、*How* を必ず記載 |
| Reviewers | *必ず 2 人以上* |
| CI ステータス | **すべて成功** でマージ許可 |
| Merge 方法 | **Rebase & Merge**（履歴を直線化） |
| コミット数 | 5 以上は *Squash* で 1 つにまとめる |
| タグ | `@team/...` で担当者タグ付け |

> **Tip**  
> PR を作成したら、必ず *CI* を再実行。  
> 失敗した場合は **「修正コミット」** を作り、再度 PR を更新。

---

## 5. コードレビュー

1. **レビュータイミング** – PR 作成後 **30 分以内** にレビュー開始を目指す。  
2. **チェックリスト**  
   - コーディング規約遵守（Lint 通過）  
   - 単体テストが増えているか  
   - 変更点がドキュメントに反映されているか  
   - セキュリティ（入力検証・権限チェック）  
   - パフォーマンスへの影響  
3. **コメントは建設的に** – 「〜なぜ？」より「〜の代わりにこうしよう」  
4. **Approve までの承認数** – **2**（1 つは *リード*、もう 1 つは *QA*）  

---

## 6. CI/CD 連携

| フェーズ | 主要ジョブ |
|----------|------------|
| **Build** | 依存関係解決、コンパイル、Lint |
| **Test** | 単体テスト・統合テスト |
| **Static Analysis** | SonarQube/CodeQL |
| **Deploy** | Staging → Production (Canary/Blue‑Green) |

- **自動マージ**: `develop` への PR が **CI 全パス成功** 時に自動マージ。  
- **自動リリース**: `main` へのマージで Docker イメージを Build → ECR → ECS/Fargate。  

---

## 7. タグとリリース管理

- **Git Tag** は `vMAJOR.MINOR.PATCH`（SemVer）で付与。  
- `git tag -a v1.2.0 -m "Release 1.2.0"`  
- タグは **リリース後に** `git push origin --tags` で公開。  
- 変更点は **CHANGELOG.md** に追記。  
- リリースノートは自動生成ツール（`conventional-changelog`）で管理。  

---

## 8. トラブルシューティング

| 状況 | 原因 | 対応 |
|------|------|------|
| **マージコンフリクト** | 大量の並行変更 | `git rebase` で `develop` を最新化し、コンフリクトを解消 |
| **CI が失敗** | 依存バージョン違い | Dockerfile のキャッシュを無効化、`pip install --upgrade` を実行 |
| **レビュープロセスが遅い** | レビュアー不足 | **レビュアー割り当て** を自動化 (GitHub Actions) |
| **ブランチが汚染** | 長期ブランチ | `git reflog expire --expire=now --all && git gc --prune=now` でクリーンアップ |

---

## 9. まとめ

- **ルールは書くだけでなく、日々の開発フローに組み込む**  
- **レビューと CI を組み合わせることで品質を自動保証**  
- **ブランチ戦略を明確にすることで衝突を減らし、デプロイをスムーズに**  
- **チーム全員が共通理解を持つ**ことで、メンテナンスコストを大幅に削減できます。

チームで Git をより効率的に使うために、まずは「現在の運用を洗い出し、必要なルールを少しずつ導入」することから始めてみましょう。共通の基盤ができれば、エンジニアは「コードを書く」ことに集中できるようになります。 🚀
