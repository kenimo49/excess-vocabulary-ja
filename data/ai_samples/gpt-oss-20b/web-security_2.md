# Webアプリケーションのセキュリティ対策

## はじめに  
近年、Webアプリケーションは企業業務の中心に位置していますが、同時に攻撃対象としても人気です。  
「**脆弱性は一歩先に見極める**」という姿勢で開発を進めることが、信頼性とビジネス継続性を保つ鍵となります。  
この記事では、エンジニアの方が実際に手を動かしやすい観点から、代表的な脆弱性と対策をまとめました。

---

## 1. セキュリティ対策の基礎  
| 項目 | 目的 | 実装例 |
|------|------|--------|
| **入力検証** | 予期しないデータを排除 | `strtolower` / `preg_match` |
| **出力エスケープ** | XSS を防止 | `htmlspecialchars()` (PHP) |
| **セッション管理** | セッションハイジャック対策 | `session_regenerate_id()` |
| **認証・認可** | アクセス制御 | OAuth2 / JWT |
| **通信暗号化** | データ盗聴対策 | HTTPS, TLS 1.3 |

---

## 2. 代表的な脆弱性と対策

### 2.1 XSS（クロスサイトスクリプティング）
* **対策**: 出力時に必ずエスケープ。信頼できない入力は**サニタイズ**。  
```php
// 入力: <script>alert(1)</script>
$escaped = htmlspecialchars($_GET['q'], ENT_QUOTES, 'UTF-8');
echo $escaped; // &lt;script&gt;alert(1)&lt;/script&gt;
```

### 2.2 CSRF（クロスサイトリクエストフォージェリ）
* **対策**: CSRF トークンをリクエスト毎に生成し、検証。  
```html
<form method="post">
  <input type="hidden" name="csrf_token" value="<?= $token ?>">
  …
</form>
```

### 2.3 SQLインジェクション
* **対策**: **プリペアドステートメント**を使用。  
```java
PreparedStatement ps = con.prepareStatement(
    "SELECT * FROM users WHERE id = ?");
ps.setInt(1, userId);
ResultSet rs = ps.executeQuery();
```

### 2.4 パスワード管理
* **対策**: ハッシュ化＋ソルト。  
```python
import bcrypt
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt())
```

### 2.5 ファイルアップロード
* **対策**: MIME タイプチェック、拡張子の制限、アップロードディレクトリをWebルート外に配置。

## 3. 開発プロセスでの取り入れ方

1. **要件定義**  
   - OWASP Top 10 を参照し、対象アプリに関係する項目を洗い出す。

2. **設計フェーズ**  
   - **入力検証ルール**を仕様書化。  
   - **CSRF・XSS** 対策を UI フローに組み込む設計。

3. **実装**  
   - コードレビュー時に「入力 → 検証 → エスケープ」のサイクルが確実に実装されているか確認。  
   - 既存フレームワークの **自動化機能**（例: Rails の `protect_from_forgery`）を積極的に活用。

4. **テスト**  
   - **自動化テスト**：SAST（静的解析）＋動的テスト（Pen Test）を併用。  
   - `jest` + `@testing-library/react` で入力検証のユニットテストを実装。

5. **運用**  
   - **WAF** の導入や **DDoS** 防御を含むインフラレベルの対策。  
   - 定期的な **脆弱性スキャン**（OWASP ZAP、Burp Suite）を CI/CD に組み込む。

---

## 4. まとめ  
- **入力検証** と **出力エスケープ** は基本中の基本。  
- OWASP Top 10 をフレームワーク化し、**開発全体に浸透**させる。  
- ただ対策を「導入」するだけでは不十分。**テスト・レビュー・運用**で継続的に検証することが重要です。

セキュリティは「最後に追加するもの」ではなく、**設計・実装・運用のすべてに組み込む文化**です。  
この記事をヒントに、日々の開発に少しでも安全性を加えていきましょう。
