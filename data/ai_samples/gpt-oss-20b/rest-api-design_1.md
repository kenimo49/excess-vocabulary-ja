# REST API設計のベストプラクティス

エンジニアの皆さん、APIを設計するとき「どうしたらシンプルで拡張しやすい構造になる？」と悩むことはありませんか？  
ここでは実務でよく使われる REST API の設計原則と、失敗しやすい落とし穴をまとめました。実装段階にすぐ落とし込める「チェックリスト」感覚で読んでください。

---

## 1️⃣ まずはリソース設計を見直す

| 項目 | 具体例 | ポイント |
|------|--------|----------|
| **エンティティ** | `/users`, `/orders` | 「何を扱うか」を明確に。単数形ではなく複数形が慣例です。 |
| **サブリソース** | `/users/123/orders` | 親子関係を表す。階層は 2〜3 段階程度に留める。 |
| **属性** | `/orders/456/status` | 属性ごとのリソース化は「状態を変える」操作に限定。 |

> **Tip**  
> なるべく URL は「読むだけで意味が分かる」ように。`/users?status=active` のようにクエリで絞り込むと、リソース名は変わらないので管理しやすい。

---

## 2️⃣ HTTP メソッドは意図通りに使う

| メソッド | 意味 | 典型的なリクエスト |
|----------|------|------------------|
| **GET** | 読み取り | `/products/42` |
| **POST** | 新規作成 | `/orders` |
| **PUT** | 完全更新 | `/users/5` |
| **PATCH** | 部分更新 | `/users/5` |
| **DELETE** | 削除 | `/items/99` |

> **ポイント**  
> `PUT` は「上書き」なので同一リクエストは idempotent。`PATCH` は「差分」を送る形にし、クライアントは変更したいフィールドだけを含める。

---

## 3️⃣ 適切なステータスコードを返す

| コード | 使い方 | 例 |
|--------|--------|----|
| 200 OK | 成功（GET, PUT, PATCH） | `{"id":1,"name":"Alice"}` |
| 201 Created | 新規作成成功 | `Location: /orders/123` |
| 204 No Content | 削除・更新成功（ボディ不要） |  |
| 400 Bad Request | 不正なリクエスト | `{"error":"invalid email"}` |
| 401 Unauthorized | 認証未完了 |  |
| 403 Forbidden | 権限不足 |  |
| 404 Not Found | リソースが存在しない |  |
| 500 Internal Server Error | サーバ側の不具合 |  |

> **Tip**  
> エラーレスポンスは必ず JSON で統一し、`error`, `message`, `code` などの構造を決めておくとフロント側が扱いやすい。

---

## 4️⃣ バージョニングで後方互換性を保つ

| 方法 | 例 | メリット |
|------|----|----------|
| **URL にバージョン番号** | `/v1/users` | シンプルに管理できる |
| **ヘッダーでバージョン** | `Accept: application/vnd.myapi.v1+json` | クライアント側にバージョンを明示させる |
| **クエリパラメータ** | `/users?api_version=1` | 柔軟だが、キャッシュが効きにくい |

> **注意**  
> 既存クライアントを壊さないために、旧バージョンは **できるだけ長く残す**。バージョンを上げる際は `deprecated` フィールドで古い API を通知しておくと良い。

---

## 5️⃣ ページング・フィルタリング・ソート

| 要素 | 実装例 | 重要ポイント |
|------|--------|--------------|
| **ページング** | `?page=2&per_page=20` | `Link` ヘッダーで `next`, `prev` を返す |
| **フィルタ** | `?status=active` | 直感的に使えるキーワードを選ぶ |
| **ソート** | `?sort=created_at,desc` | 主要なカラムを前提にしておく |

> **Tip**  
> 大量データは `cursor` ベースで取得する方がパフォーマンス向上するケースが増えている。例: `?cursor=eyJpZCI6MTIzNH0=`（Base64 エンコード）。

---

## 6️⃣ 認証・認可は「OAuth 2.0」や「JWT」を活用

- **Bearer Token** を `Authorization` ヘッダーで送る。  
- `scope` でリソース単位の権限を付与し、**最小権限原則**を徹底。  
- スコープが不十分なリクエストは 403 で拒否。

> **実装例**  
> ```http
> GET /users/123
> Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6...
> ```

---

## 7️⃣ テスト・監視・ドキュメントを並行して行う

| 項目 | ツール | 目的 |
|------|--------|------|
| **テスト** | Postman, REST Assured, Swagger‑Hub | スキーマと挙動の自動検証 |
| **監視** | Prometheus, Grafana | レイテンシ・エラーレートの可視化 |
| **ドキュメント** | OpenAPI (Swagger), RAML | API をコードと同時にバージョン管理 |

> **Tip**  
> OpenAPI を使えば自動生成されたクライアントライブラリで、フロントエンド開発が高速化します。

---

## 8️⃣ 失敗しやすい落とし穴

| 落とし穴 | 回避策 |
|----------|--------|
| **状態管理を URL に隠す** | ステータスはレスポンスボディで返す |
| **非 idempotent で PUT/PATCH を使う** | POST を使うか、明示的に「更新済み」フラグを設ける |
| **過剰なネスト** | `/users/123/orders/456/items` より、`/orders/456/items` のほうがシンプル |

---

## まとめ

- **リソースは名付けを明確に**  
- **HTTP メソッドとステータスコードを正しく使う**  
- **バージョン管理を計画的に**  
- **ページング・フィルタ・認証を統一的に設計**  
- **テスト・監視・ドキュメントを並行実装**

設計は「やり直す」よりも「見直す」方がコストが低いものです。  
この記事を参照しながら、まずは既存の API をレビューし、次に新規開発でこれらの原則を取り入れてみてください。  

Happy coding! 🚀
