# REST API設計のベストプラクティス ― エンジニア向けガイド

REST（Representational State Transfer）は、Webサービスを設計する際に最も広く採用されているアーキテクチャスタイルです。  
「リソースをURLで表現し、HTTPメソッドで操作する」だけでなく、**一貫性**・**拡張性**・**保守性**を意識した設計が重要です。  
この記事では、実際に開発・運用を行うエンジニアに役立つベストプラクティスを整理します。

---

## 1. リソース設計 ― URL は“集合体”であるべき

| ✅ | ❌ |
|----|----|
| `/users`  | `/getUserById?id=123` |
| `/users/123` | `/users?id=123` |

- **集合体（Collection）** を `/users`、**個別リソース** を `/users/123` と表現する。  
- **複数形** を使用して自然言語に近づける。  
- ルートは **シンプル** かつ **意味的に一貫** するように設計。  
- URLに処理名を入れる（`/users/search`）は避け、クエリパラメータでフィルタリングを行う。

---

## 2. HTTPメソッドは意味に合わせて使う

| メソッド | 典型的な用途 | ステータスコード例 |
|-----------|--------------|-------------------|
| GET | 取得 | 200 OK / 404 Not Found |
| POST | 作成 | 201 Created |
| PUT | 完全更新 | 200 OK / 204 No Content |
| PATCH | 部分更新 | 200 OK / 204 No Content |
| DELETE | 削除 | 200 OK / 204 No Content |

- **PUT** は完全な置換を意味する。  
- **PATCH** は変更したフィールドのみを送信。  
- `DELETE` でリソースが存在しない場合は **404**、ただし削除が冪等であれば **204** が推奨。

---

## 3. ステータスコードは「何が起きたか」を明示

| ステータスコード | 意味 |
|------------------|------|
| 200 OK | 成功。レスポンスボディに結果を返す。 |
| 201 Created | リソース作成成功。`Location`ヘッダーで新規URIを返す。 |
| 204 No Content | 成功だがボディは不要。 |
| 400 Bad Request | リクエストが不正。 |
| 401 Unauthorized | 認証情報がない／無効。 |
| 403 Forbidden | 権限不足。 |
| 404 Not Found | 対象が存在しない。 |
| 409 Conflict | 競合。 |
| 429 Too Many Requests | レート制限。 |
| 500 Internal Server Error | サーバ側の障害。 |

---

## 4. バージョニング ― スキーマ変更への対策

- **URLバージョニング**（`/v1/users`）が最も単純で実装しやすい。  
- 変更が頻繁な API では **クエリパラメータ**（`?api_version=2`）も併用可能。  
- バージョン管理は**非破壊的**に行い、古いバージョンは**徐々に廃止**する。  
- バージョン切替時には **リリースノート** を用意し、顧客に通知する。

---

## 5. ページング・フィルタリング・ソート

| パラメータ | 例 | 説明 |
|-----------|----|------|
| `page` | `?page=2` | ページ番号 |
| `per_page` | `?per_page=20` | 1ページあたりの件数 |
| `sort` | `?sort=-created_at` | ソートキー（`-`は降順） |
| `filter` | `?status=active` | フィルタ条件 |

- **ページング**は**offset**でなく**cursor**（次のカーソル）方式が高速。  
- **リンクヘッダー**（`Link: <...>; rel="next"`）で次ページを明示。  
- フィルタは**単一キー**で、複雑なクエリは**検索エンドポイント**を設ける。

---

## 6. HATEOAS（Hypermedia as the Engine of Application State）

- レスポンスに**リンク**を埋め込み、クライアントが次に行える操作を示す。  
- 例:  
  ```json
  {
    "id": 123,
    "name": "Alice",
    "_links": {
      "self": { "href": "/users/123" },
      "orders": { "href": "/users/123/orders" }
    }
  }
  ```
- 実装は必須ではないが、**自動化されたクライアント生成**や**ドキュメント整合性**を担保したい場合に有効。

---

## 7. エラー表現 ― 標準化と情報量

- **JSON API** 仕様（`application/vnd.api+json`）に沿ったエラー構造を採用。  
  ```json
  {
    "errors": [
      {
        "status": "400",
        "title": "Invalid Attribute",
        "detail": "The 'email' field must be a valid email address."
      }
    ]
  }
  ```
- `title` で簡潔に、`detail` で具体的な説明。  
- `source` で該当フィールドを指摘するとデバッグが楽になる。  

---

## 8. 認証・認可 ― それぞれ分離する

- **OAuth 2.0 / OpenID Connect** が標準。  
- **Bearer トークン**を`Authorization`ヘッダーに付与。  
- スコープ（`scope`）でリソースごとの権限を細かく設定。  
- **レートリミット**は認可レベル別に設計し、**429**で通知。

---

## 9. キャッシュ戦略

| ヘッダー | 用途 |
|-----------|------|
| `Cache-Control` | キャッシュポリシー（`public`, `private`, `no-store` 等） |
| `ETag` | リソース変更検知 |
| `Last-Modified` | 変更日時 |
| `If-None-Match` / `If-Modified-Since` | 条件付きリクエスト |

- **GET** は基本的にキャッシュ可能。  
- **POST/PUT/PATCH/DELETE** はキャッシュ除外。  
- CDN を活用し、**静的リソース**は長期間キャッシュ。

---

## 10. ドキュメント化 ― Swagger/OpenAPI で自動生成

- **OpenAPI 3.0** をベースに定義し、**UI**（Swagger‑UI / ReDoc）で可視化。  
- 変更時に **CI/CD** で検証（`swagger-cli validate`）し、**テスト**と**ドキュメント**を同時に更新。  
- 受け取るクライアント側で **TypeScript** 型自動生成 (`openapi-typescript`) を利用するとミスを減らせる。

---

## 11. テスト戦略

1. **単体テスト** – ルーティング、バリデーション、ビジネスロジックを個別に確認。  
2. **統合テスト** – 実際のエンドポイントへリクエストを投げ、**HTTP** レスポンスを検証。  
3. **パフォーマンステスト** – 負荷・レイテンシを測定し、キャッシュやレートリミットを調整。  
4. **セキュリティテスト** – OWASP Top 10（XSS, CSRF, Injection 等）を自動化。

テストは**CI パイプライン**で必ず走らせ、**回帰防止**を徹底。

---

## 12. 運用 ― モニタリングとロギング

- **OpenTelemetry** でトレーシングを実装し、**稼働状況**・**遅延**を可視化。  
- **Structured Logging**（JSON）で**検索性**と**統合**を実現。  
- **アラート**は**レスポンスコード**・**レイテンシ**・**レートリミット**を指標に設定。  

---

## まとめ

REST APIは「単なるデータ転送」ではなく、**使いやすさ・拡張性・安全性**を同時に満たす設計が求められます。  
上記のベストプラクティスを順守すれば、以下のメリットが期待できます。

| メリット | 具体例 |
|-----------|--------|
| **可読性** | `/users/123` で直感的にリソースを把握。 |
| **メンテナンス性** | バージョンを切り替えても旧クライアントが影響を受けない。 |
| **開発スピード** | Swagger/OpenAPI で自動生成したクライアントがすぐに使える。 |
| **品質保証** | ルール化されたステータスコード・エラーレスポンスでテスト容易。 |
| **安全性** | OAuth 2.0 + レートリミットで攻撃を抑制。 |

設計の段階から **ベストプラクティス** を意識して実装すれば、長期にわたる API の品質と信頼性を確保できます。ぜひ、プロジェクトに取り入れてみてください。
