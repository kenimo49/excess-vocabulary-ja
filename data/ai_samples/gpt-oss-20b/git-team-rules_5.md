# Gitチーム運用ルール ― エンジニア向け実践ガイド

## 1. 目的  
Gitは分散型であるが、チーム開発では「統一した運用ルール」が品質と生産性を左右します。本記事では、エンジニアが直感的に実装できる具体的なルールを紹介します。チームの規模やプロジェクトによって微調整してください。

## 2. 基本ルール

| 項目 | ルール | 目的 |
|------|--------|------|
| **ブランチ戦略** | `main`/`master` は常にデプロイ可能状態を保ち、`dev`（もしくは`develop`）は最新の開発ブランチにする。<br>機能追加は `feature/<名前>`、バグ修正は `bugfix/<番号>`、リリースは `release/<番号>` | 変更の可視化とリリース管理を容易にする |
| **ブランチ名** | スネークケースで短く、意味が明確。例: `feature/user-auth`、`bugfix/12-issue` | 検索性と可読性を向上 |
| **コミットメッセージ** | 1行目は50文字以内で要約、2行目は空行、3行目以降は詳細を記述。<br>例: `Add JWT auth middleware\n\nAdd a middleware that verifies JWT tokens for protected routes.` | チェンジログの自動生成や検索が容易になる |
| **Pull Request（PR）** | PR は必ず 1 つのタスクを完結させる。<br>レビューは 2 人以上が必ず行い、承認済みでなければマージ不可。<br>PR のタイトルはブランチ名と同じで OK。 | バグを早期発見し、コード品質を保つ |
| **レビュー基準** | ・機能実装が要件を満たしているか。<br>・テストが網羅的か。<br>・リファクタリングが適切か。<br>・セキュリティ上の懸念がないか。 | コードベースの一貫性と安全性を確保 |
| **テスト** | 単体・結合テストを必ず書き、CI で自動実行。<br>テストカバレッジは 80% 以上を目標。 | バグの再発防止とリファクタリングの安全性を高める |
| **CI/CD** | PR がマージされるたびに自動テスト・デプロイを実行。<br>環境ごとに `infra/<環境名>` ブランチを持ち、変更は `infra/feature-<name>` から PR。 | デプロイ失敗のリスクを低減 |

## 3. 詳細設定例

### 3.1. Git のフックを使った自動検証

```bash
#!/bin/sh
# .git/hooks/commit-msg
if ! grep -qE '^(feat|fix|docs|style|refactor|test|chore):' "$1"; then
  echo "❌ Commit message must start with a type (feat, fix, etc.)"
  exit 1
fi
```

このフックはコミットメッセージが慣例に沿っているかをチェックし、違反した場合はコミットを拒否します。

### 3.2. Git Flow の簡易化

```bash
# 機能ブランチ作成
git checkout -b feature/user-auth develop

# バグ修正
git checkout -b bugfix/12-issue develop

# リリースブランチ
git checkout -b release/1.0.0 develop
```

`develop` は `main` の前段階として使用し、リリース前にテストを完結させます。

## 4. チーム文化としての取り組み

1. **コードレビューを歓迎**：ネガティブな言葉は避け、改善点は具体的に示す。  
2. **ドキュメントを更新**：PR でコードが変わるたびに README やコメントも更新。  
3. **定期的なリファクタリング**：技術的負債を先延ばしにしない。  
4. **コミュニケーション**：Slack 等で「#dev」チャネルに PR URL を共有し、随時フィードバックを受ける。

## 5. まとめ  

Git の運用ルールは「定める」だけでなく、**実際に使う中でフィードバックを取り入れ、改善していく**ことが重要です。  
上記のルールを導入すれば、以下のメリットが期待できます。

- コミット履歴が読みやすくなる → バグ追跡が楽になる  
- PR でのレビューが統一的に行える → 品質が安定する  
- CI/CD がスムーズ → デプロイ失敗が減る  

チームのスケールに合わせて微調整し、みんなが安心してコードを書ける環境を整えてください。 Happy coding! 🚀
