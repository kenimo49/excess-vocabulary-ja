# Git チーム運用ルール ― 共同開発を円滑にするためのベストプラクティス

## はじめに
組織での Git 利用は「個人開発」ではなく「チーム開発」が前提です。  
ただ単にリポジトリを共有しているだけでは、ブランチ管理の不備やコミットメッセージの不統一などで衝突が増え、リリース遅延やバグが発生しやすくなります。  
本記事では、エンジニアが実際に手順を守りやすい「Git チーム運用ルール」を整理します。文字数は 1000‑2000 文字程度に収め、実践的な例を交えつつ Markdown でまとめます。

---

## 1. ブランチ戦略を決める
### 1‑1. 主要ブランチ
| ブランチ | 目的 | 例 |
|----------|------|-----|
| `main`（あるいは `master`） | 本番リリース済みコード | `main` |
| `develop` | 次期リリースの統合 | `develop` |
| `feature/*` | 機能追加 | `feature/login-flow` |
| `bugfix/*` | バグ修正 | `bugfix/login-error` |
| `release/*` | リリース前準備 | `release/1.2.0` |
| `hotfix/*` | 本番緊急修正 | `hotfix/1.2.1` |

> **ポイント**  
> * `main` は常にデプロイ可能な状態に保つ。  
> * `develop` は `main` の最新を取り込みつつ、開発中の機能を集約。  
> * `feature/*` は短期的に完結できる単位で切る。

### 1‑2. ブランチ作成・マージ手順
1. **ローカルでブランチ作成**  
   ```bash
   git checkout -b feature/login-flow develop
   ```
2. **作業完了後にコミット**  
3. **リモートにプッシュ**  
   ```bash
   git push -u origin feature/login-flow
   ```
4. **Pull Request（PR）を作成**  
   * 目的・変更点を記載  
   * 必要に応じてレビュワーを指定
5. **CI が成功し、レビューが完了したらマージ**  
   * `squash merge` でコミット数を整理  
   * マージ後にローカルを `develop` に反映  
     ```bash
     git checkout develop
     git pull
     ```

## 2. コミットメッセージ規約
統一されたメッセージは後から変更点を追いやすく、GitHub の「変更履歴」を活用しやすくします。  
以下は **Conventional Commits** 形式をベースにした例です。

```
<type>(<scope>): <subject>

<body>

<footer>
```

| type | 説明 |
|------|------|
| feat | 新機能 |
| fix | バグ修正 |
| docs | ドキュメントのみ |
| style | コードフォーマット等 |
| refactor | リファクタ |
| test | テスト追加/修正 |
| chore | ビルド/ツール |

> **例**  
> `feat(login): OAuth 認証を追加`

---

## 3. コードレビュー & Pull Request のルール
* **レビュワーは最低 1 名**。  
* **レビュー時間**：5 分以内にコメントを返す。  
* **レビュー項目**  
  - コードの可読性（コメント・変数名）  
  - テストカバレッジ  
  - セキュリティリスク  
  - パフォーマンスへの影響  
* **CI パイプライン**  
  - 静的解析 (`eslint`, `clang-tidy` など)  
  - ユニットテスト (`jest`, `pytest` など)  
  - ビルド成功確認  
* **マージ前のチェック**  
  - **ブランチ**：`feature/*` → `develop`  
  - **コミット数**：1～2 本に絞る  
  - **差分レビュー**：コードの変更点が明確に分かるように

---

## 4. タグとリリース
1. `develop` がリリース候補になったら `release/*` ブランチを作成。  
2. バージョン番号は **SemVer** (MAJOR.MINOR.PATCH) を採用。  
3. タグ付けはリリース完了時に `main` へマージしつつ行う。  
   ```bash
   git tag -a v1.2.0 -m "Release 1.2.0"
   git push origin v1.2.0
   ```

---

## 5. Issue とタスク管理
* **Issue** には必ず「カテゴリ」「優先度」「担当者」をタグ付け。  
* **マイルストーン** を作成し、 `feature/*` や `bugfix/*` で進捗管理。  
* PR のタイトルに Issue 番号を付与すると自動でリンクが貼られる。  
  ```
  feat: add new API endpoint (#123)
  ```

---

## 6. コーディング規約と共通ライブラリ
* プロジェクト共通の **ESLint / Prettier** 設定をルートに置き、`.gitignore` で `node_modules` などを除外。  
* 共通ロジックは **ライブラリ** へ切り出し、`npm publish` でバージョン管理。  
* 依存関係は `package-lock.json` / `yarn.lock` を必ずコミット。

---

## 7. まとめ
| 規約 | 目的 |
|------|------|
| ブランチ戦略 | コンフリクトを最小化 |
| コミット規約 | 変更履歴の可読性向上 |
| コードレビュー | 品質保証と知識共有 |
| タグ・リリース | 正確なバージョン管理 |
| Issue 管理 | 進捗把握と優先順位付け |
| コーディング規約 | 一貫したコードベース |

実際に **チーム内で同意を得たルール** を書面化し、リポジトリの README に掲載しておくと新人も即座に作業に入れます。  
ルールは「完璧」ではなく「実践可能」であることが重要です。  
継続的にフィードバックを取り入れ、チームの開発フローに合わせて微調整しましょう。
