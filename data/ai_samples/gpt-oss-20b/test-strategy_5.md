# ソフトウェアテスト戦略の立て方 ― エンジニア向け実践ガイド

## 1. なぜテスト戦略が必要か  
テストは「コードの品質を保証する」だけでなく、  
* **リスク低減** – 重大なバグを早期発見  
* **リリース速度** – 自動化を前提にした継続的デリバリー  
* **コスト最適化** – 失敗したテストを無駄にしない  
を実現します。戦略を持たないまま開発を進めると、後でテストケースが膨大になり、修正コストが急増します。

## 2. 戦略設計の基本フロー

| ステップ | 目的 | 具体的なアウトプット |
|----------|------|------------------------|
| **① 要件・リスク分析** | 何をテストすべきかを定義 | リスクマトリクス、機能一覧 |
| **② テストレイヤ設計** | テストの粒度と自動化レベルを決定 | テストパイロット図、手動/自動比率 |
| **③ テストケース作成** | 実装に直結したケースを作る | テストケーステンプレート、データセット |
| **④ 自動化戦略** | CI/CDに統合した自動化フロー | テストスイート構成、フレームワーク選定 |
| **⑤ 運用と改善** | 実際の運用を通じて改善 | メトリクス定義、レビューサイクル |

## 3. テストパイオニアモデル（Pyramid）

```
┌───────────────────────────────┐
│ ① E2E（UI）  – 10%（手動）     │
├───────────────────────────────┤
│ ② インテグレーション  – 30%（自動）│
├───────────────────────────────┤
│ ③ ユニット  – 60%（自動）       │
└───────────────────────────────┘
```

* **ユニット**：高速で失敗の原因を局所化  
* **インテグレーション**：依存関係を検証  
* **E2E**：エンドユーザー視点で検証（頻度は低め）

## 4. リスクベーステストの実装

1. **リスク特定**  
   * 重要度 × 影響度  
   * 例：認証フローの不備 → 重大
2. **テスト優先度付与**  
   * リスクが高い箇所を最優先で自動化  
   * 低リスクは手動＋簡易テスト
3. **定期的にリスク再評価**  
   * 新機能追加時に更新

## 5. 自動化戦略の設計

| フレームワーク | 主な用途 | 代表例 |
|----------------|----------|--------|
| Jest / Mocha   | ユニットテスト | JavaScript |
| JUnit          | インテグレーション | Java |
| Cypress / Playwright | E2E | Web アプリ |

### CI/CDパイプライン例（GitHub Actions）

```yaml
name: CI

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: [14, 16]
    steps:
      - uses: actions/checkout@v3
      - name: Use Node.js
        uses: actions/setup-node@v3
        with:
          node-version: ${{ matrix.node-version }}
      - run: npm ci
      - run: npm test
```

## 6. メトリクスで戦略を可視化

| 指標 | 目的 | 目安 |
|------|------|------|
| **カバレッジ** | テスト網羅率 | 80% 以上（ユニット） |
| **デフォルト率** | 失敗率 | < 5% |
| **平均修正時間** | 修正効率 | 1〜2営業日 |
| **テスト実行時間** | CI 時間 | < 5 分 |

## 7. テスト戦略ドキュメントのテンプレート

```markdown
# テスト戦略

## 1. 目的
- 何を達成するか

## 2. テストレイヤ
| レイヤ | 比率 | 手動/自動 | フレームワーク |
|--------|------|-----------|----------------|
| ユニット | 60% | 自動 | Jest |
| インテグレーション | 30% | 自動 | JUnit |
| E2E | 10% | 手動 | Cypress |

## 3. リスクマトリクス
| 機能 | 重要度 | 影響度 | 優先度 |
|------|--------|--------|--------|

## 4. 自動化方針
- コミット毎にテストを実行
- PR 時に E2E を実行

## 5. メトリクス
- カバレッジ：80%
- 失敗率：<5%
```

## 8. まとめ  
テスト戦略は「テストのやり方」だけでなく、**リスク評価・自動化レベル・運用サイクル**を統合したロードマップです。  
1️⃣ 要件からリスクまで把握し、  
2️⃣ パイロット図でレイヤを決定し、  
3️⃣ CI で継続的に実行・改善する、  
この３段階を意識すれば、品質と速度を両立した開発が実現できます。

ぜひ、自社のプロジェクトに合わせてカスタマイズし、実際に戦略を策定・運用してみてください。
