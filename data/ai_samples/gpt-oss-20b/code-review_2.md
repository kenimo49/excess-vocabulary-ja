# コードレビューの効果的なやり方 ― エンジニア向け実践ガイド

> **TL;DR**  
> 1. 目的を明確に → 目的を共有  
> 2. チェックリストを作成 → 何を見ているか可視化  
> 3. 30分以内に完了 → レビューのフローを整える  
> 4. フィードバックは具体的・建設的に → 「何が悪いか」ではなく「どうすればよいか」  
> 5. ツールを活用 → CI／自動化で「必ずチェック」  

---

## 1. コードレビューの目的を「品質向上」だけにしない

多くのチームでは「バグを早く見つける」ことが主目的ですが、実際には以下のような副次的効果も期待できます。

| 目的 | 具体的な効果 |
|------|--------------|
| **品質向上** | バグ検出、テストカバレッジ向上 |
| **知識共有** | アーキテクチャ理解、ベストプラクティスの浸透 |
| **チームの統一感** | コーディング規約への統一、リファクタリングの一貫性 |
| **リスク低減** | 将来の保守コスト削減、技術的負債の発見 |

**まずは目的をチーム全員で共有**し、レビュー時に何を達成したいのかを意識しましょう。

---

## 2. チェックリストで「見逃し」を減らす

レビューは「盲目的に読む」よりも「質問を投げる」方が効率的です。以下のようなチェックリストを用意しておくと、抜け漏れを防げます。

```text
- コードの可読性
  - 変数名・関数名は意味があるか？
  - コメントは必要最低限か？
- コーディング規約
  - ルール違反はないか？
  - フォーマッティングは統一されているか？
- バグ・脆弱性
  - 例外処理は網羅的か？
  - 外部入力に対して検証はあるか？
- パフォーマンス
  - O(n^2) などの低速箇所はないか？
  - 必要のない再計算はないか？
- テスト
  - 変更箇所のテストケースは追加されたか？
  - カバレッジは維持または向上したか？
- ドキュメント
  - 変更点はREADMEやコメントに反映されているか？
```

チェックリストは **PRを開く前に自分で完了**し、レビュアーは「完了済み」と見やすいマーク（例：✅）を付けると効率が上がります。

---

## 3. 「30分ルール」でレビューを短時間に完結

「時間が足りない」レビューはクオリティを下げる原因になります。実際に **30分以内にレビューを完了させる** ことを目標にしましょう。以下のポイントが鍵です。

| ポイント | 実践例 |
|----------|--------|
| **事前準備** | PR作成前に必ずチェックリストを確認。 |
| **分割** | 大きな変更は小さな PR に分割。 |
| **フォーカス** | 1 回のレビューで 1 つのテーマに集中（例：性能）。 |
| **コミュニケーション** | コメントは簡潔に。「詳細はメールで」ではなく、PR 内で完結。 |

30 分ルールを守ることで、レビュアーの負担も軽減し、**「レビューが遅延」** という問題を解消できます。

---

## 4. フィードバックは「具体的・建設的に」

> ❗ **注意**：感情的な批判は逆効果。  
> 「コードが悪い」ではなく「こうすると良い」  

### 具体的に言うためのパターン

| フィードバック | 書き方例 |
|----------------|----------|
| **ポジティブ** | *「このロジックは読みやすいですね。…」* |
| **ネガティブ** | *「ここで `ArrayList` を使っているのは良くないと思います。`LinkedList` にすると O(1) でアクセスできます。」* |
| **提案** | *「こういう実装方法もあります。`Optional` を使うと null チェックが簡潔になります。」* |

**コメントを残す際は**  
1. **まずは良い点を指摘**（モチベーション維持）  
2. **問題点を明確に**  
3. **改善策を提示**  
4. **質問で終わる**（相手に考えさせる）

## 5. ツールで自動化を活用

レビューは人間の作業ですが、**ツールで「必ずチェック」**を自動化することで効率が大幅に向上します。

| ツール | 役割 |
|--------|------|
| **GitHub Actions / GitLab CI** | コーディング規約チェック、テスト自動実行 |
| **SonarQube / CodeClimate** | 静的解析、コード品質指標 |
| **Reviewdog** | PR コメントに自動解析結果を反映 |
| **ESLint / Prettier / RuboCop** | フォーマッティング・Lint |

例:  
```yaml
# .github/workflows/code-quality.yml
name: Code Quality
on: pull_request
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Install Node
        uses: actions/setup-node@v3
        with:
          node-version: '18'
      - run: npm ci
      - run: npm run lint
      - name: Reviewdog
        uses: reviewdog/action-eslint@v1
        with:
          reporter: github-pr-review
```

こうした自動化は **「人のレビューを減らす」** ではなく、**「人が読むべきポイントを絞る」** ことに重きを置きます。

---

## まとめ

- **目的を共有**し、レビューを「品質向上＋知識共有」の両面で捉える。  
- **チェックリスト**で抜け漏れを防ぎ、**30分ルール**で短時間に完結。  
- **フィードバックは具体的・建設的**に。  
- **自動化ツール**でルール違反やバグを事前に検出し、レビュアーは本質的な改善点に集中。  

実践すれば、レビューはチームの**技術力を飛躍的に高める**武器になるはずです。ぜひ、今日から「レビュー＝価値創造」として取り組んでみてください。
