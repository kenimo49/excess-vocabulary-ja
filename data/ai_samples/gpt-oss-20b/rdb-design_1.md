# リレーショナルデータベース設計の基本

リレーショナルデータベース（RDB）は、データを表（テーブル）で管理し、**正規化**と**制約**を駆使して一貫性・可用性を確保します。本稿では、エンジニアがすぐに実務に落とし込める設計の要点をまとめます。

## 1. まずは ER モデルで全体像を可視化

1. **エンティティ**（人・商品・注文など）  
2. **属性**（名前・価格・日付など）  
3. **関係**（1対多・多対多）  

ER 図を描くことで、データ間の関係性が明確になります。例:

```
[Customer] 1 ──────< [Order] >───── 1 [Product]
```

## 2. 正規化で重複を排除

| 正規形 | 目的 | 典型的な問題 |
|--------|------|--------------|
| **1NF** | 原子性 | `phone = "090-1111-2222,080-3333-4444"` など |
| **2NF** | 部分関数従属除去 | `Order` に `product_price` を含める |
| **3NF** | 推移的従属除去 | `Customer` に `city_id` と `city_name` を同時に保持 |
| **BCNF** | すべての関数従属が候補キーに依存 | 役職名から役職コードが決まる場合 |

**実務ヒント**  
- 3NF まででほとんどの場合十分です。  
- パフォーマンス重視ならば、特定の検索パターンに合わせて**デノーマライズ**（冗長データを許容）します。

## 3. キーと制約の設計

| キー | 用途 | 注意点 |
|------|------|--------|
| **主キー（PK）** | 行の一意性 | `id` は整数（Auto Increment）で OK |
| **外部キー（FK）** | 関連テーブル整合性 | `ON DELETE CASCADE` の使用は意図的に |
| **ユニーク制約** | 重複禁止 | 例えば `email`  |
| **チェック制約** | 範囲・フォーマット | `age BETWEEN 0 AND 120` |

制約は **データ整合性をコードレベルで保証** します。アプリ側でのバリデーションは重複させないようにしましょう。

## 4. インデックス設計

| シナリオ | 推奨インデックス | 補足 |
|----------|------------------|------|
| 主キー検索 | 既に PK でインデックス済 | |
| 外部キー検索 | `FK` 列に単一インデックス | |
| 複合検索 | `WHERE colA = ? AND colB = ?` | 複合インデックスを作成 |
| ソート | `ORDER BY col DESC` | 先にソート対象列を含める |

インデックスは検索速度を劇的に向上させますが、**書き込み時のオーバーヘッド**もあるため、使用頻度を基に選択してください。

## 5. テーブル設計のチェックリスト

- [ ] ER 図があるか  
- [ ] 3NF まで正規化されているか  
- [ ] 主キー・外部キーが定義済みか  
- [ ] 必要なユニーク制約があるか  
- [ ] チェック制約でデータの妥当性が保たれているか  
- [ ] 適切なインデックスを設置しているか  

## まとめ

リレーショナルデータベース設計は **論理的な構造** と **物理的な性能** のバランスが鍵です。  
ER 図で設計を可視化し、正規化でデータ冗長を排除し、制約・インデックスで整合性とパフォーマンスを確保する――これが基本です。  
実際の開発では、ビジネス要件や運用コストを見ながら、必要に応じてデノーマライズやインデックスのチューニングを行ってください。
