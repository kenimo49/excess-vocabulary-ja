# リレーショナルデータベース設計の基本

## はじめに
リレーショナルデータベース（RDB）は、アプリケーションの基盤として広く利用されています。設計を誤ると、データの整合性が失われたり、パフォーマンスが落ちたりします。本記事では、エンジニア向けに「リレーショナルデータベース設計」の基本概念と実践的なポイントを解説します。1000〜2000字程度にまとめているので、すぐに実務に落とし込めます。

## 1. コンセプト設計（概念設計）
まずは、ビジネス要件を理解し、どのような「エンティティ（実体）」があるかを洗い出します。  
例）`User`（顧客）、`Order`（受注）、`Product`（商品）。

> **ポイント**  
> - すべてのエンティティは「主観的に意味がある」ものに限定する。  
> - 一つのエンティティに複数の役割が混在しないように注意する。

## 2. 論理設計（ER図作成）
概念設計をもとに、ER図（Entity‑Relationship Diagram）を作ります。  
- **エンティティ** → テーブル  
- **リレーション** → 外部キー  
- **属性** → カラム

### 例
```
┌─────────┐     ┌────────────┐
│  User   │◄────│  Order     │
├─────────┤     ├────────────┤
│ id PK   │     │ id PK      │
│ name    │     │ user_id FK │
└─────────┘     │ total      │
                 └────────────┘
```

## 3. 正規化
データの重複を防ぎ、一貫性を保つために正規化を行います。  
| 正規形 | 主な目的 | 例 |
|--------|----------|---|
| 1NF    | 重複行をなくす | 同一顧客は同じ`User`行に |
| 2NF    | 部分関数従属を排除 | `Order`の`product_name`は`Product`に移動 |
| 3NF    | 推移的従属を排除 | `Order`の`total`は計算列に |

> **BCNF**（ベイツ正規形）はさらに厳格ですが、実務では3NFで十分な場合が多いです。

## 4. キー設計
- **主キー（PK）**：テーブル内で一意になる列。自動採番（`SERIAL`/`BIGINT`）が一般的。  
- **外部キー（FK）**：他テーブルの主キーを参照。参照整合性を保証する。  

### 制約
- **UNIQUE**：重複不可。  
- **NOT NULL**：必須。  
- **CHECK**：値の範囲制御。  
- **DEFAULT**：デフォルト値。

## 5. インデックス
クエリの高速化に不可欠。  
- **主キーは自動でユニークインデックス**が作られる。  
- **検索頻度の高い列**に対してB‑Treeインデックスを設定。  
- **複合キー**に対しては**コンビネーションインデックス**を検討。  

> **注意**  
> インデックスは書き込みコストを増やすので、必要最小限に。  

## 6. パフォーマンスと設計のトレードオフ
正規化により冗長性を排除すると、JOIN が増えパフォーマンスが落ちるケースも。  
- **デノーマライズ**（冗長列を保持）で読み取りを高速化する。  
- **キャッシュ**（Redis など）を併用し、頻繁に参照されるデータを分離。  

## 7. まとめ
1. ビジネス要件からエンティティを洗い出し、ER図を作成。  
2. 1NF〜3NFまで正規化し、主外キー・制約でデータ整合性を確保。  
3. 読み取りパフォーマンスを考慮し、必要に応じてデノーマライズやインデックスを設計。  

設計段階での丁寧な設計は、後のバグ修正やスケールアウトを大幅に簡略化します。実務では「完璧」より「実用的」で「拡張可能」な設計を心がけてください。
