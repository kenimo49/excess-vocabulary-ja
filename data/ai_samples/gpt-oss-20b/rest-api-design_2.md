# REST API設計のベストプラクティス

エンジニアの皆さん、RESTful API を設計する際に意識したいポイントをまとめました。  
この記事は、実務でよく直面する課題を解決するための「手順・ルール」を整理しています。  
(文字数: 約1400字)

---

## 1. リソース設計の基本

| 項目 | 推奨 | 例 |
|------|------|----|
| **URIは複数形** | `/users`, `/orders` | ① /user → ② `/users` |
| **階層構造は意味を持たせる** | `/users/123/orders` | ユーザー 123 の注文 |
| **状態遷移はHTTPメソッドで表現** | `PUT /orders/123/ship` → ① 受注 → ② 出荷状態に変更 | ③ `/orders/123/ship` でリソースを更新 |
| **副作用のないGET** | `/users/123` は読み取り専用 | ④ /users/123?include=orders |

> **ポイント**  
> - リソースは *集合* として扱う（`/users`）  
> - 個別リソースは *URI* の末尾に ID を置く  
> - 「状態遷移」もリソースとみなすと一貫性が保てる

---

## 2. HTTPメソッドとステータスコード

| メソッド | 役割 | 推奨ステータスコード |
|----------|------|------------------------|
| GET | 取得 | 200 OK, 304 Not Modified |
| POST | 作成 | 201 Created |
| PUT | 完全更新 | 200 OK, 204 No Content |
| PATCH | 部分更新 | 200 OK, 204 No Content |
| DELETE | 削除 | 204 No Content, 404 Not Found |

> **注意点**  
> - POST で作成した際は `Location` ヘッダーに新リソース URI を返す  
> - エラー時は **4xx** 系で詳細メッセージを JSON で返す  
> - 成功時に必ず `Content-Type: application/json` を明示

## 3. ページング・フィルタリング

| 技術 | 実装例 |
|------|--------|
| **ページング** | `?page=2&limit=20`、または `?offset=40&limit=20` |
| **オフセット vs キー** | 大規模データは「スキップ」を避け、`?after=<last_id>` で前後検索 |
| **フィルタ** | `?status=active&created_before=2024-01-01` |
| **ソート** | `?sort=-created_at`（降順） |

> **ベストプラクティス**  
> - **リンクベース** (RFC 5988) を活用し、`Link: <...>; rel="next"` を返す  
> - 返すデータ量は 1 リクエスト 1 〜 2 KB 以内に抑える  
> - `limit` の上限は API ドキュメントで明記し、サーバ側で検証

---

## 4. バージョニング

| 方法 | 特徴 |
|------|------|
| **URL バージョニング** | `/v1/users` |
| **ヘッダー** | `Accept: application/vnd.myapi.v1+json` |
| **クエリパラメータ** | `?api_version=1` |

> **推奨**  
> - 変更が大きい場合は URL バージョニング  
> - 互換性を保ちつつ拡張したい場合はヘッダーでバージョン管理  
> - バージョンを廃止する際は 3 か月以上のデプリケーション期間を設ける

---

## 5. エラーハンドリング

```json
{
  "errors": [
    {
      "status": "400",
      "title": "Bad Request",
      "detail": "email is required",
      "source": { "pointer": "/data/attributes/email" }
    }
  ]
}
```

- **JSON:API** 形式を採用すると統一感が出る  
- `errors` 配列で複数エラーを返せる  
- **ステータスコード** は最も適切なものを返す（例：400, 404, 409, 422）

## 6. セキュリティ

| 項目 | 推奨 |
|------|------|
| 認証 | OAuth 2.0 / JWT |
| 権限 | RBAC / Attribute-based |
| 暗号化 | HTTPS/TLS で通信を暗号化 |
| レートリミット | `/users` などエンドポイントごとに設定 |
| CORS | 必要最低限のオリジンのみ許可 |

> **ポイント**  
> - 認証情報は `Authorization` ヘッダーに送る  
> - アクセス権限は *最小権限* 原則で設計  
> - エラーメッセージに機密情報は漏らさない

---

## 7. ドキュメント

- **OpenAPI（Swagger）** で仕様を自動生成  
- **自動生成されたサンプルリクエスト/レスポンス** を含める  
- **バージョン管理** を Git で行い、リリースごとに `v1.0.yaml` を保存  
- **変更ログ** を必ず追記（Changelog）

---

## 8. テスト戦略

| 方向 | テスト内容 |
|------|------------|
| **単体** | モジュール単位でレスポンスフォーマットを検証 |
| **統合** | 実際の HTTP リクエストでエンドポイントを走査 |
| **負荷** | ページング・レートリミットの挙動を確認 |
| **セキュリティ** | OWASP ZAP などで脆弱性チェック |

> **ベストプラクティス**  
> - テストは **CI/CD** で自動化  
> - カバレッジは 90% 以上を目指す  
> - エラーパスも必ずテスト

---

## まとめ

1. **リソース名は複数形・階層的**  
2. **HTTP メソッドとステータスコードを正しく使う**  
3. **ページング・フィルタはリンクベースで実装**  
4. **バージョンは明確に**  
5. **エラーは JSON:API で統一**  
6. **認証・権限は最小権限で**  
7. **OpenAPI でドキュメント化**  
8. **CI/CD でテスト自動化**  

これらを実践することで、**拡張性・保守性・ユーザビリティ**を兼ね備えた REST API を構築できます。
