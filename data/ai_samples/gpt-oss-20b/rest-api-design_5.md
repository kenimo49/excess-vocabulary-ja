# REST API設計のベストプラクティス  
> **対象読者**：エンジニア（バックエンド／フロントエンド・両方）  
> **文字数**：約1,500文字  

---

## 1. はじめに  
REST API は「HTTP/HTTPS でやり取りするサービス」の代表格です。設計を誤ると、クライアント側の実装が複雑化し、保守性が低下します。この記事では、実務で何度も試行錯誤した「ベストプラクティス」をまとめ、設計時のチェックリストとして活用してもらいます。

---

## 2. リソース指向の設計  
### 2-1. URI はリソースを表す  
- **集合**：`/books`  
- **個体**：`/books/123`  
- **子リソース**：`/books/123/authors`  

リソースは **名詞** で表現し、**状態** でなく **操作** は HTTP メソッドに委ねる。

### 2-2. 冗長なパラメータは排除  
クエリ文字列は検索・フィルタに限定。  
```text
GET /books?author=山田&sort=price
```
過剰に多いパラメータはエンドポイントを増やすか、**GraphQL** を検討。

---

## 3. HTTP メソッドの正しい使い分け  
| メソッド | 目的 | 推奨使用ケース |
|----------|------|----------------|
| `GET` | 読み取り | `/books/123` |
| `POST` | 作成 | `/books` |
| `PUT` | 完全更新 | `/books/123` |
| `PATCH` | 部分更新 | `/books/123` |
| `DELETE` | 削除 | `/books/123` |

> **ポイント**  
> - `PUT` はリソース全体を置き換える想定。  
> - `PATCH` は JSON Merge Patch を推奨。  

---

## 4. ステータスコードの一貫性  
| コード | 意味 | 典型的な例 |
|--------|------|------------|
| `200 OK` | 正常取得 | `GET /books/123` |
| `201 Created` | 作成完了 | `POST /books` |
| `204 No Content` | 削除/更新成功でレスポンス不要 | `DELETE /books/123` |
| `400 Bad Request` | バリデーションエラー | `POST /books` で必須項目欠如 |
| `401 Unauthorized` | 認証未完了 | 認証ヘッダー未送信 |
| `403 Forbidden` | 権限不足 | 管理者専用エンドポイント |
| `404 Not Found` | リソース未存在 | `/books/9999` |
| `409 Conflict` | 一意制約違反 | 同名書籍作成 |

> **ベストプラクティス**  
> - **必ず** ステータスコードを返す。  
> - エラー時は `application/json` で `error` フィールドを返す。

```json
{
  "error": {
    "code": "ERR_INVALID_PARAMETER",
    "message": "Title is required."
  }
}
```

---

## 5. ページング・フィルタリング  
大量データを返す場合は必ず **ページング** を実装。  
```text
GET /books?page=2&size=20
```
- `page` は 1 ベース推奨。  
- `size` は上限 100 を設ける。  

レスポンス例  
```json
{
  "items": [ /* 20 件 */ ],
  "page": 2,
  "size": 20,
  "totalCount": 135
}
```

**サポート**  
- `Link` ヘッダーで `next`, `prev` を提供。  
- **Cursor‑based** ページング（`?cursor=abc123`）を検討するとパフォーマンス向上。

---

## 6. エラー処理とロギング  
- **エラーコード** は内部で統一し、**メッセージ** はローカライズ可能に。  
- **サーバー側ログ** は **JSON** で出力し、Kibana などで可視化。  
- **監視**：Sentry/Datadog で 5xx をアラートに。

---

## 7. バージョニング  
- **URL** にバージョンを含める：`/v1/books`  
- 変更が大きい場合は **非互換** バージョンを新設。  
- **ヘッダー** でバージョン指定（`Accept: application/vnd.myapp.v1+json`）は、後方互換性が必須なケースに。  

---

## 8. セキュリティ  
| 項目 | 実装例 |
|------|--------|
| 認証 | JWT（Authorization: Bearer） |
| 認可 | RBAC/ABAC でリソース単位 |
| TLS | HTTPS 強制 |
| Rate‑limit | `X-RateLimit-Limit`, `X-RateLimit-Remaining` ヘッダーで制御 |
| CORS | 必要なオリジンのみ許可、`Access-Control-Allow-Credentials` で Cookie 連携 |

---

## 9. ドキュメント & テスト  
- **OpenAPI（Swagger）** でスキーマを自動生成。  
- **自動テスト**：Postman/Newman, Schemathesis で契約テスト。  
- **CI/CD** で **ドキュメント生成** をパイプラインに組み込み、常に最新版を提供。

---

## 10. まとめ  
| ✅ | チェック項目 |
|---|--------------|
| リソース指向 | URI は名詞で表現 |
| メソッド一貫 | GET/POST/PUT/PATCH/DELETE を正しく使う |
| ステータスコード | 1xx–5xx を適切に返す |
| ページング | `page`/`size` で制御 |
| エラー処理 | JSON で詳細を返し、ロギング |
| バージョン | `/vX/` で管理 |
| セキュリティ | HTTPS, JWT, RBAC |
| ドキュメント | OpenAPI + CI/CD |

設計段階でこのチェックリストを参照すれば、クライアントが扱いやすく、メンテナンスもしやすい REST API を構築できます。ぜひプロジェクトに取り入れてみてください！
