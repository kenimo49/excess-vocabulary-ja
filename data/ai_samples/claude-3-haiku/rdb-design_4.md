# リレーショナルデータベース設計の基本

## はじめに

アプリケーションを開発する上で、データを管理するためのデータベースの設計は非常に重要です。特にリレーショナルデータベースは、多くのアプリケーションで広く使われているデータベース方式です。この記事では、リレーショナルデータベース設計の基本について解説していきます。エンジニアの方を対象としていますので、データベース設計の基本的な考え方や手法について、具体的な例を交えながら説明していきます。

## リレーショナルデータベースとは

リレーショナルデータベースは、データを表形式で管理するデータベース方式です。データを「テーブル」と呼ばれる二次元の表形式で保持し、テーブル間の関係性(リレーション)を定義することで、データの整合性を保ちながらデータを管理することができます。

主な特徴は以下の通りです。

- データをテーブル形式で管理する
- テーブル間の関係性(リレーション)を定義できる
- SQL(構造化問い合わせ言語)を使ってデータの検索・更新ができる
- トランザクション処理による整合性の維持

リレーショナルデータベースには、MySQL、PostgreSQL、Oracle Database、Microsoft SQL Serverなど、様々な製品が存在します。これらのデータベース製品はSQLに対応しており、SQLを使ってデータの操作を行うことができます。

## データベース設計の基本

### エンティティの特定
データベース設計の最初のステップは、アプリケーションで扱うデータのエンティティを特定することです。エンティティとは、データベースで管理するデータの単位を表すものです。
例えば、Webショッピングサイトを考えると、主なエンティティとして「商品」「注文」「ユーザー」などが考えられます。

### 属性の洗い出し
次に、各エンティティがどのような属性(フィールド)を持つかを洗い出します。属性には、エンティティを特徴づける情報が含まれます。
例えば、「商品」エンティティであれば、「商品名」「商品説明」「価格」「在庫数」などの属性が考えられます。

### 主キーの設定
各エンティティには、一意に識別できる主キーを設定する必要があります。主キーは、そのエンティティのレコードを一意に識別する属性です。
通常は、自動採番される整数型の ID カラムを主キーとして設定します。これにより、レコードの追加や更新、削除が容易に行えます。

### エンティティ間の関係性の定義
エンティティ間の関係性を定義します。リレーショナルデータベースでは、1対1、1対多、多対多の関係を定義できます。
例えば、「ユーザー」と「注文」の関係は1対多となります。1人のユーザーが複数の注文をする関係を表すことができます。

### 正規化
正規化は、データの冗長性を排除し、データの整合性を保つための手法です。データベース設計では、できるだけ正規化された形で設計を行うことが重要です。
正規化には、第1正規形、第2正規形、第3正規形など、いくつかのレベルが存在します。データベース設計では、状況に応じて適切な正規化レベルを選択する必要があります。

### インデックスの設計
クエリ性能を向上させるため、適切なインデックスを設計する必要があります。主キーや外部キーには、自動的にインデックスが作成されますが、それ以外の属性についても、検索頻度が高いものにはインデックスを設定します。

### データ型の設定
各属性のデータ型を適切に設定する必要があります。データ型の選択は、データの性質や使用目的に合わせて行います。例えば、数値データであれば整数型や浮動小数点型を、テキストデータであれば文字列型を選択するなどです。

## 具体的な設計例

ここでは、Webショッピングサイトのデータベース設計例を示します。

### エンティティ定義
- ユーザー
  - ユーザーID
  - 名前
  - メールアドレス
  - パスワード
- 商品
  - 商品ID
  - 商品名
  - 商品説明
  - 価格
  - 在庫数
- 注文
  - 注文ID
  - ユーザーID
  - 商品ID
  - 注文数量
  - 注文日時

### エンティティ間の関係性
- ユーザーと注文は1対多の関係
- 商品と注文は1対多の関係

### テーブル設計
上記のエンティティに基づいて、以下のようなテーブル設計が考えられます。

**users テーブル**
| カラム名 | データ型 | 主キー | 
| --- | --- | --- |
| id | INT(11) | ○ |
| name | VARCHAR(50) | |
| email | VARCHAR(255) | |
| password | VARCHAR(100) | |

**products テーブル** 
| カラム名 | データ型 | 主キー |
| --- | --- | --- |
| id | INT(11) | ○ |
| name | VARCHAR(100) | |
| description | TEXT | |
| price | DECIMAL(10,2) | |
| stock | INT(11) | |

**orders テーブル**
| カラム名 | データ型 | 主キー |
| --- | --- | --- |
| id | INT(11) | ○ |
| user_id | INT(11) | |
| product_id | INT(11) | |
| quantity | INT(11) | |
| ordered_at | TIMESTAMP | |

### 正規化
上記のテーブル設計は、第3正規形に準拠しています。ユーザーテーブルと商品テーブルには、それぞれ主キーが設定されており、注文テーブルではユーザーテーブルと商品テーブルへの外部キー参照が行われています。

### インデックスの設計
主キーには自動的にインデックスが設定されますが、それ以外にも検索頻度の高いカラムにインデックスを設定することで、クエリ性能の向上が期待できます。例えば、以下のようなインデックスが考えられます。

- `users` テーブル: `email` カラムにインデックスを設定
- `products` テーブル: `name` カラムにインデックスを設定
- `orders` テーブル: `user_id` カラムと `product_id` カラムにインデックスを設定
