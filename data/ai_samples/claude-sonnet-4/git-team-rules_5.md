# Gitチーム運用ルール：効率的な共同開発のためのベストプラクティス

## はじめに

チーム開発において、Gitの運用ルールを明確に定めることは、プロジェクトの成功に不可欠です。適切なルールがなければ、コンフリクトの頻発、コードレビューの困難、リリース作業の複雑化など、様々な問題が発生します。本記事では、実際のチーム開発で効果的なGit運用ルールについて解説します。

## ブランチ戦略

### Git Flow を基本とした運用

多くのチームで採用されているGit Flowをベースとしたブランチ戦略を推奨します。

```
main (production)
├── develop (開発統合)
├── feature/ISSUE-123-user-authentication (機能開発)
├── release/v1.2.0 (リリース準備)
└── hotfix/critical-security-fix (緊急修正)
```

**各ブランチの役割：**

- `main`: 本番環境にデプロイされるコード
- `develop`: 開発中の最新コードを統合
- `feature/*`: 機能開発・バグ修正用
- `release/*`: リリース準備用
- `hotfix/*`: 本番環境の緊急修正用

### ブランチ命名規則

統一された命名規則により、ブランチの目的が一目で分かります。

```bash
# 機能開発
feature/ISSUE-番号-簡潔な説明
# 例: feature/ISSUE-456-payment-integration

# バグ修正
bugfix/ISSUE-番号-簡潔な説明
# 例: bugfix/ISSUE-789-login-error

# ホットフィックス
hotfix/緊急度-簡潔な説明
# 例: hotfix/critical-sql-injection
```

## コミット運用ルール

### コミットメッセージの規約

[Conventional Commits](https://www.conventionalcommits.org/) 形式を採用します。

```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

**主要なタイプ：**

- `feat`: 新機能
- `fix`: バグ修正
- `docs`: ドキュメント変更
- `style`: コードフォーマット変更
- `refactor`: リファクタリング
- `test`: テスト追加・修正
- `chore`: ビルド・設定ファイル変更

**例：**

```bash
feat(auth): ユーザー認証機能を追加

- JWT トークンベースの認証を実装
- ログイン・ログアウト API を追加
- セッション管理機能を実装

Closes #123
```

### コミット粒度のガイドライン

- **1つの論理的変更 = 1つのコミット**
- 関連のない変更は別々のコミットに分割
- コミット単位でテストが通る状態を保持
- レビュー時に理解しやすい適切なサイズを心がける

## プルリクエスト（PR）運用

### PR作成時のルール

1. **テンプレートの活用**

```markdown
## 変更概要
<!-- 何を変更したかを簡潔に記述 -->

## 変更理由
<!-- なぜこの変更が必要かを説明 -->

## テスト内容
- [ ] 単体テスト
- [ ] 結合テスト
- [ ] 手動テスト

## チェックリスト
- [ ] コードレビュー完了
- [ ] テスト追加・修正
- [ ] ドキュメント更新

## 関連Issue
Closes #123
```

2. **ドラフトPRの活用**

開発途中でも積極的にドラフトPRを作成し、早期フィードバックを得る文化を推奨します。

### レビュー運用ルール

- **必須レビュアー数**: 最低2名（シニア1名＋同レベル以上1名）
- **レビュー期限**: 営業日2日以内
- **レビューの観点**:
  - ビジネスロジックの正確性
  - パフォーマンス
  - セキュリティ
  - 可読性・保守性
  - テストカバレッジ

## マージ戦略

### Squash and Merge を推奨

```bash
# 機能ブランチ → develop
Squash and Merge を使用

# develop → main (リリース時)
Merge commit を使用してリリース履歴を保持
```

この戦略により、mainブランチの履歴をクリーンに保ちながら、必要に応じて詳細な開発履歴も追跡可能です。

## リリース運用

### セマンティックバージョニング

```
MAJOR.MINOR.PATCH (例: v2.1.3)
```

- **MAJOR**: 破壊的変更
- **MINOR**: 新機能追加（後方互換性あり）
- **PATCH**: バグ修正

### リリースフロー

1. `develop` から `release/vX.Y.Z` ブランチを作成
2. リリース準備作業（バージョン更新、CHANGELOG作成等）
3. QAテスト実施
4. `main` と `develop` にマージ
5. タグ作成とリリースノート公開

## 禁止事項とトラブル対応

### 絶対にやってはいけないこと

- `main` ブランチへの直接push
- 公開済みコミットの `git rebase` や `git reset`
- 大きすぎるPR（500行以上の変更）
- 意味のないコミットメッセージ（"fix", "update" のみ等）

### トラブル時の対応

**コンフリクト発生時：**

```bash
# developの最新を取得
git checkout develop
git pull origin develop

# featureブランチでrebase
git checkout feature/your-branch
git rebase develop

# コンフリクト解決後
git rebase --continue
git push --force-with-lease origin feature/your-branch
```

## まとめ

効果的なGitチーム運用ルールは、開発効率の向上とコード品質の担保に直結します。重要なのは、チーム全体でルールを理解・共有し、継続的に改善していくことです。

新メンバーのオンボーディング時には、これらのルールを丁寧に説明し、実践を通じて習得してもらいましょう。また、プロジェクトの成長に合わせて、運用ルールも柔軟に見直していくことが大切です。

適切な運用ルールにより、ストレスフリーな開発環境を実現し、チーム全体の生産性向上を目指しましょう。
